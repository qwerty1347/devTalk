# 03 EC2(외부 서버) 배포

> 참고용 문서. devTalk의 기본 배포는 **Vercel(서버리스)** 이다([02](02%20CICD-배포.md)).
> 이 문서는 EC2 같은 **상시 서버**에 배포하는 전통적 방식을 정리한 것으로,
> SSH 배포·서버 운영을 학습하거나 직접 서버를 운영해야 할 때 참고한다.

## 서버리스와의 핵심 차이

| | Vercel (서버리스) | EC2 (서버) |
|--|-------------------|-----------|
| 앱 실행 | 요청 시에만 함수 실행 | `next start`로 상시 실행 |
| 서버 준비 | 없음 | EC2 직접 생성·관리 |
| 프로세스 관리 | 자동 | pm2로 직접 |
| 웹서버 | 자동 | nginx 직접 |
| HTTPS | 자동 | certbot 직접 |
| 환경변수 | 대시보드 | 서버의 `.env.local` |
| 배포 | `git push`만 | SSH 배포 잡 + Secrets |
| 비용 | 무료 티어(안 쓰면 0원) | 상시 과금(켜져 있는 동안) |

## 전체 흐름

```
git push
   │
   ▼
GitHub Actions
   ├─ build/typecheck (검사 — 서버리스와 동일)
   └─ deploy: SSH로 EC2 접속
        → git pull → npm ci → build → pm2 restart
                                          │
                                          ▼
                                EC2에서 next start 상시 실행
                                (nginx: 80 → 3000 프록시)
```

## 1. EC2 최초 셋업 (한 번만)

EC2에 SSH 접속 후:

```bash
# Node.js (nvm)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
nvm install 20

# pm2 (Node 프로세스 상시 관리)
npm i -g pm2

# 코드 클론 & 빌드
cd /var/www
git clone https://github.com/qwerty1347/devTalk.git
cd devTalk
npm ci
npm run build

# 실행 (pm2가 next start 관리)
pm2 start "npm run start" --name devTalk
pm2 save
pm2 startup        # 재부팅 시 자동 실행
```

→ EC2의 `3000` 포트에서 앱이 돈다.

## 2. nginx 리버스 프록시 (80 → 3000)

```bash
sudo apt install nginx
```

`/etc/nginx/sites-available/devTalk`:
```nginx
server {
    listen 80;
    server_name 내도메인.com;   # 또는 EC2 퍼블릭 IP

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```
```bash
sudo ln -s /etc/nginx/sites-available/devTalk /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

> HTTPS는 `certbot`(Let's Encrypt)으로 무료 인증서 적용.

## 3. 환경변수 (서버에 직접)

Vercel과 달리 서버의 `.env.local`에 직접 둔다(이 파일은 git에 없음):
```bash
cd /var/www/devTalk
nano .env.local
# GEMINI_API_KEY=...
# PINECONE_API_KEY=...
# ... 등
```

## 4. GitHub Actions에 deploy 잡 추가

기존 `ci.yml`(검사)에 SSH 배포 잡을 덧붙인다:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build

  deploy:
    needs: build              # 검사 통과해야 배포
    runs-on: ubuntu-latest
    steps:
      - name: EC2에 SSH 배포
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ${{ secrets.EC2_USER }}
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            cd /var/www/devTalk
            git pull origin main
            npm ci
            npm run build
            pm2 restart devTalk
```

## 5. GitHub Secrets 등록

저장소 → Settings → Secrets and variables → Actions → New secret:

| Secret | 값 |
|--------|-----|
| `EC2_HOST` | EC2 퍼블릭 IP (예: 13.125.x.x) |
| `EC2_USER` | `ubuntu` 또는 `ec2-user` |
| `EC2_SSH_KEY` | EC2 `.pem` 키 파일 내용 전체 |

## 완성된 흐름

```
git push → Actions 검사 → 통과 → SSH로 EC2 접속
        → git pull·build·pm2 restart → 사이트 갱신
```

## 주의

- EC2는 **안 써도 켜져 있는 동안 계속 과금**된다. 연습은 **프리티어(t2.micro, 1년 무료)** 권장.
- 서버·nginx·HTTPS·보안그룹(포트 22/80/443 오픈) 등 직접 관리할 게 많다.
- devTalk의 실제 운영은 Vercel이 더 간단하고 저렴하다. 이 문서는 **학습/특수 상황용** 참고.
